# Event Dictionary — MDT Phase 1 POC

**Version:** 3.0  
**Status:** Refactored / Aligned  

---

## 1. Purpose

Defines all events generated by the MDT system during a shift.

These events are the single source of truth for:
- activity logging
- timesheet generation
- shift summary
- future analytics

---

## 2. Design Principles

- one user action may generate multiple events
- append-only
- consistent structure
- self-contained enough for local reconstruction
- all Phase 1 events originate from MDT

---

## 3. Core Entities

### 3.1 Shift Session

| Field | Type | Description |

## 4. Core Events

- `SHIFT_STARTED`
- `ACTIVITY_STARTED`
- `ACTIVITY_ENDED`
- `SHIFT_ENDED`

`ACTIVITY_SWITCHED` is excluded in Phase 1.

---

## 5. Event Catalog

### 5.1 SHIFT_STARTED

Triggered when operator submits Start Shift.

Payload:
- `eventId`
- `eventName = SHIFT_STARTED`
- `shiftSessionId`
- `unitId`
- `operatorId`
- `hmStart`
- `occurredAt`
- `source`

Side effects:
- create active shift session
- automatically start default activity using a separate `ACTIVITY_STARTED`

---

### 5.2 ACTIVITY_STARTED

Triggered when an activity becomes active.

Payload:
- `eventId`
- `eventName = ACTIVITY_STARTED`
- `shiftSessionId`
- `unitId`
- `operatorId`
- `activityCategory`
- `activitySubtype`
- `occurredAt`
- `source`
- `loaderCode` (required if subtype = `loading`)
- `haulingCode` (required if subtype = `hauling`)

Rules:
- `loaderCode` is alphanumeric
- `haulingCode` is alphanumeric
- `dumping` has no conditional input
- no location fields in Phase 1

---

### 5.3 ACTIVITY_ENDED

Triggered when the current activity stops.

Payload:
- `eventId`
- `eventName = ACTIVITY_ENDED`
- `shiftSessionId`
- `unitId`
- `operatorId`
- `activityCategory`
- `activitySubtype`
- `occurredAt`
- `source`

---

### 5.4 SHIFT_ENDED

Triggered when operator confirms End Shift.

Payload:
- `eventId`
- `eventName = SHIFT_ENDED`
- `shiftSessionId`
- `unitId`
- `operatorId`
- `hmEnd`
- `occurredAt`
- `source`

Side effects:
- if an activity is active, `ACTIVITY_ENDED` is generated first
- shift session becomes `ended`

## 7. Event Sequence Rules

### Rule 1 — Shift must start first
Valid:
```text
SHIFT_STARTED → ACTIVITY_STARTED
```

### Rule 2 — Default activity starts immediately
After `SHIFT_STARTED`, system must generate:
```text
ACTIVITY_STARTED (standby, changeShift)
```

### Rule 3 — Single active activity
There can only be one unmatched `ACTIVITY_STARTED` at a time.

### Rule 4 — Activity switch sequence
```text
ACTIVITY_ENDED → ACTIVITY_STARTED
```

### Rule 5 — End Shift sequence
```text
ACTIVITY_ENDED → SHIFT_ENDED
```

### Rule 6 — Shared switch timestamp
The end event and new start event share the same timestamp at the switch point.

## 8. Derived Fields

### Activity-level
- `durationSeconds = endedAt - startedAt`
- `deltaHm = derivedHmEnd - derivedHmStart`

### Shift-level
- total operation seconds
- total standby seconds
- total delay seconds
- total breakdown seconds
- total shift seconds
- total delta HM
- PA
- UA

### Count Derivation
- `loadingCount` = number of `ACTIVITY_STARTED` with subtype `loading`
- `haulingCount` = number of `ACTIVITY_STARTED` with subtype `hauling`

Counts are cumulative and derived.

- no external integrations
- local-only storage
- no edit/delete events
